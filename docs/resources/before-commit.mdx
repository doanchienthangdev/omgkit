# Before-Commit Checklist

OMGKIT provides two types of before-commit rules:

1. **OMGKIT Internal Rules** - For contributing to OMGKIT itself
2. **Project Rules** - For projects developed with OMGKIT

---

## Rules Location

| Rule Type | Location | Purpose |
|-----------|----------|---------|
| **OMGKIT Internal** | `plugin/stdrules/` | OMGKIT development only |
| **Project Rules** | `.omgkit/stdrules/BEFORE_COMMIT.md` | User projects |

---

## For OMGKIT Contributors

If you're contributing to OMGKIT itself, follow the internal rules.

### Quick Validation

```bash
npm test
```

All 4800+ tests must pass before committing.

### Internal Rule Files

- `plugin/stdrules/ALIGNMENT_PRINCIPLE.md` - Component hierarchy rules
- `plugin/stdrules/OMGKIT_BEFORE_COMMIT_RULES.md` - Commit validation rules
- `plugin/stdrules/SKILL_STANDARDS.md` - Skill documentation standards

### Documentation Quality Standards

#### Agent Documentation
- 50+ lines of content
- Valid YAML frontmatter with: `name`, `description`, `tools`, `skills`, `commands`

#### Command Documentation
- 15+ lines of content
- Valid YAML frontmatter with: `description`, `allowed-tools`

#### Skill Documentation
- 30+ lines of content
- YAML frontmatter with: `name`, `description`

#### Workflow Documentation
- 50+ lines of content
- Valid YAML frontmatter with: `description`, `agents`

### Command Registration

Every command MUST be registered as a Claude Code slash command:

```
plugin/commands/<namespace>/<command-name>.md
```

Format: `/namespace:command-name` (lowercase, hyphens for word separation)

### Registry Integrity

- Version must match `package.json`
- Minimum: 30+ agents, 100+ commands, 100+ skills, 40+ workflows
- All cross-references must exist

### Prohibited Content

| Pattern | Description |
|---------|-------------|
| `TODO:` | Actual placeholder at line start |
| `FIXME:` | Actual placeholder at line start |
| `HACK:` | Actual placeholder at line start |
| `[INSERT ...]` | Placeholder brackets |
| `[PLACEHOLDER]` | Placeholder markers |

### Validation Test Files

| File | Purpose |
|------|---------|
| `tests/validation/before-commit.test.js` | Master validation suite |
| `tests/validation/doc-quality.test.js` | Documentation quality |
| `tests/validation/command-registration.test.js` | Slash commands |
| `tests/validation/registry-sync.test.js` | Registry consistency |
| `tests/validation/alignment.test.js` | Alignment principle |

## For Project Developers

If you're developing a project with OMGKIT, use the project rules.

### Getting Started

When you initialize a project with `omgkit init`, a `BEFORE_COMMIT.md` file is created in `.omgkit/stdrules/`.

### Project Rule Categories

1. **Code Quality**
   - Tests pass
   - No linting errors
   - Build succeeds

2. **Git Standards**
   - Conventional commits
   - Meaningful messages
   - Clean branch hygiene

3. **Security**
   - No secrets in code
   - Dependencies audited
   - Input validation

4. **Documentation**
   - README updated
   - CHANGELOG maintained
   - API docs current

5. **Performance**
   - No regressions
   - Assets optimized

6. **Review Checklist**
   - Self-review done
   - Edge cases covered
   - Error handling in place

### Quick Project Check

```bash
npm test && npm run lint && npm run build
```

### Customization

The `.omgkit/stdrules/BEFORE_COMMIT.md` file includes a section for project-specific rules. Add your custom requirements there.

## Commit Standards

Both OMGKIT and projects should use conventional commits:

```bash
git commit -m "type(scope): description"
```

| Type | Description |
|------|-------------|
| `feat` | New feature |
| `fix` | Bug fix |
| `docs` | Documentation |
| `chore` | Maintenance |
| `refactor` | Code restructuring |
| `test` | Adding tests |
| `perf` | Performance |
| `ci` | CI/CD changes |
